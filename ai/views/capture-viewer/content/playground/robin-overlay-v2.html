shield Settings Enforcement
System-wide write-lock preventing AI from modifying any folder named settings/. This is the primary human-in-the-loop gate — the AI can read settings but never write to them directly.
Status Active
Type Hardcoded
Source lib/enforcement.js

Enforcement Active
Disable to allow AI write access to settings/ folders

How It Works

Every write_file and edit_file tool call passes through checkSettingsBounce() in the server's wire message handler. The function splits the file path into segments and tests each one against the pattern /^\.?settings$/i.

If any segment matches, the tool call is bounced — the server sends a tool_result with isError: true back to the wire, and the AI sees a restriction message. A system:tool_bounced event fires on the event bus for auditing.

What's Protected

  • settings/ — standard folder name
  • .settings/ — dot-prefixed variant
  • Settings/, .Settings/ — case variants
  • Any depth: ai/panels/code/chat/settings/PROMPT.md
  • All files inside, not just PROMPT.md and SESSION.md

Deploy Flow

Since the AI can't write to settings/ directly, it drops generated files into the parent folder. A TRIGGERS.md-driven file watcher detects the new file and shows a drag-to-deploy modal overlay. The user physically drags the file into the settings/ target to activate it. 

AI generates PROMPT.md → drops in chat/
  → trigger fires → modal appears
  → user drags to settings/
  → prior copy archived → new file deployed

Configuration

This rule is hardcoded and not configurable through TRIGGERS.md or any settings file. The override toggle above writes to system.jsonc — if you disable it, all settings/ folders become writable by AI. 

{
  "enforcement": {
    "settings_write_lock": true    // ← this toggle
  }
}
