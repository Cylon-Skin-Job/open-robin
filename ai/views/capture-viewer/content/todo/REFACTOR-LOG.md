### SPEC-01b — Startup Orchestrator

- **Observation:** Line estimate was significantly low (~150 estimated, 197 actual). The three broadcast callbacks (`sendChatMessage`, `broadcastModal`, `broadcastFileChange`) are preserved verbatim per Gotcha #7 and add ~47 lines by themselves. **Lesson for future specs:** when a region has verbatim-preserved callback trios or similar structural expansions, estimate at +50% over raw line count.
- **Behavior shift:** Signal handlers (SIGTERM / SIGINT) used to register at module load time in server.js. In `lib/startup.js`, they now register *inside* `start()` after `server.listen()` resolves. There is a sub-millisecond window between module require and listen completion where a signal would fall through to default termination instead of running `closeDb()`. **Harmless in practice** — during that window `initDb` hasn't finished, so there's nothing to clean up — but it IS a timing shift worth knowing about if someone later sees a rare "db not flushed on fast SIGTERM" log.
- **Judgment call (executing session):** `xargs kill -TERM` instead of `kill -TERM $(lsof -ti:3001)`. zsh was embedding literal `\n` into the argument and kill rejected it. Mechanically equivalent, cleaner for multi-PID. Accepted.

### What made the workflow work

1. **Explicit "what not to do" at every temptation point.** Every spec called out the refactors the executing session might be tempted to make (DRY'ing the wire-spawn sequence, converting function declarations to const expressions, splitting the router into sub-files, "fixing" the mutable-reference pattern). Naming the temptation is how you prevent it.
2. **Grep invariants with expected match counts.** "`grep clientMsg.type server.js` should return 0 matches" is unambiguous. The executing session runs the grep and knows immediately whether they're done.
3. **Gotchas with failure modes.** Not "watch out for X" but "if you do X naively, Y will happen at runtime because Z". The why lets them judge edge cases the spec didn't foresee.
4. **Exact line numbers as landmarks + drift-check instructions.** Spec says "line 304"; executing session runs `rg -n 'createWireLifecycle' server.js`; if it's at 306 they know something shifted by 2, not that their grep is wrong.
5. **A canary test that exercises the full new code path.** `restart-kimi.sh` + server-live.log scan + a headless boot were the canary for every spec. Live WS traffic (when present) was bonus proof.
6. **Single-concern commits.** One spec = one commit. No "while I'm in here" drive-bys. The extraction is the extraction.

### Estimation bias — confirmed

Line-count estimates ran **+10–20% low** consistently across all six specs. Drivers: comment symmetry around new `require` statements, multi-line factory construction vs. one-liner estimates, verbatim JSDoc transcription of spec examples, and preserved-verbatim callback trios. **Going forward:** estimate the raw extracted region + 15% buffer. Don't count JSDoc headers from spec examples toward the estimate; they're always transcribed.
